Wie bereits in Kapitel \ref{MAAllgemeines} erwähnt, wurde für dieses Projekt Flutter als Framework gewählt. Das hat mehrere Gründe, doch zunächst etwas zur Geschichte von Flutter.\\
Flutter wurde 2016 von Google auf den Markt gebracht. Im Gegensatz zu React Native, das auf einem bereits vorher bestandenen Framework aufgebaut wurde und ein Jahr früher auf den Markt gekommen ist, hat Google Flutter neu aufgebaut. Hierbei haben die Entwickler - im Gegensatz zu React Native, das in JavaScript geschrieben wird - Dart als Sprache gewählt. Das war allerdings nicht von Anfang an klar, da vor allem zu Beginn JavaScript eigentlich die erste Wahl gewesen wäre\cite{reg406}.\\

\paragraph {Dart} \label{MAFramework_Dart}

Dart wurde ursprünglich von Google für Webdevelopment entwickelt und war als Alternative zu JavaScript angedacht. Da teilweise JavaScript-Standards eingehalten worden sind, gibt es ähnliche Befehle in den beiden Sprachen wie zum Beispiel \glqq async \grqq{}oder \glqq await \grqq. Trotzdem ist die Syntax eher Java-ähnlich aufgebaut\cite{reg417}.\\
Einer der Vorteile von Dart ist, dass die Sprache AOT (ahead of time) und JIT (Just in Time) kompiliert werden kann. Code, der JIT kompiliert wird, ist oft viel langsamer, dafür ist AOT kompilierter Code nicht plattformunabhängig. Code, der JIT kompiliert wird, läuft in der Dart-VM. Mithilfe davon können Programme nur teilweise neu kompiliert werden, ohne dass das gesamte Programm neu kompiliert wird. Das ermöglicht eine viel schnellere und angenehmere Entwicklung, da Änderungen im Programm sehr schnell übertragen und ausgetestet werden können. Außerdem ermöglicht es einige Debug-Funktionen. Für die praktische Anwendung eines Projekts, nach der Entwicklungsphase, kann Dart dann in nativen Code umgewandelt werden\cite{reg417}.\\
Ein weiterer Vorteil, den Dart gegenüber JavaScript hat, ist ein generational garbage collector.\\
Während Zuteilung und auch das Freigeben von Speicher in Sprachen wie C manuell geregelt werden muss, gibt es dafür in höheren Sprachen automatisierte Prozesse, wie zum Beispiel den Garbage Collector (GC). Die Aufgabe eines GCs ist es, unbenutzten Speicher wieder freizugeben. Zu diesem Zweck wird überprüft, welche Objekte nicht mehr referenziert werden und diese dann gelöscht, wie in Abbildung \ref{fig:gr_MAFramework_GC1} zu sehen ist:

Es dauert jedoch relativ lang, eine gesamte Anwendung nach referenzierten und nicht-referenzierten Objekten zu durchsuchen. Ein Ansatz, um diesen Prozess schneller zu machen, ist der generational GC. Diese Art von GC baut darauf auf, dass die meisten Objekte sehr nah an ihrere Erstellung wieder collected werden können, und Objekte, die schon einige Zeit lang bestehen, oft noch länger bestehen als neue. Die Überlegung ist also, dass ältere Objekte nicht mehr so oft auf ihre Referenzierung überprüft werden müssen. In Abbildung \ref{fig:gr_MAFramework_GC2} ist ein möglicher Aufbau eines generational GCs gezeigt:

	\caption{verschiedene Generationen des GC\cite{reg419}}
	\label{fig:gr_MAFramework_GC2}
\end{figure}

Neue Objekte kommen hier nach Eden. Bei jedem kleineren GC-Event wird Eden und beide survivor spaces auf aussortierbare Objekte durchsucht.

Wie in Abbildung \ref{fig:gr_MAFramework_GC3} zu sehen ist, werden referenzierte Objekte in einen der survivor spaces umgelagert und unreferenzierte Objekte als neuer Speicherplatz freigegeben. Bei jedem GC-Event werden auch noch referenzierte Objekte aus einem survivor space in den jeweils anderen geschoben. Wenn diese nun ein gewisses \glqq Alter \grqq{}erreicht haben, werden sie als ältere Generation festgelegt, und nicht mehr bei jedem GC-Event sondern nur noch gelegentlich überprüft, wie in Abbildung \ref{fig:gr_MAFramework_GC4} gezeigt\cite{reg419}:

Ein weiterer Vorteil des Dart-GC ist, dass GC-Events durchgeführt werden, wenn eine App gerade untätig ist; das sorgt dafür, dass die Performance nicht durch das Event gestört wird\cite{reg421}.
Da in Flutter potenziell sehr viele Objekte und Widgets erstellt werden, die nach einer kurzen Zeit nicht mehr referenziert werden, ist der GC ein wichtiger Bestandteil.\\
Ein weiterer Vorteil Dart gegenüber JavaScript ist, dass Dart type-safe ist. Das bedeutet, wenn zum Beispiel eine Variable als Integer definiert wird, diese Variable aber ein String als Wert zugewiesen wird, dann resultiert das in einem compile-time-error (Der Code wird nicht kompiliert). Das hat mehrere Vorteile: Erstens werden somit mit dem Typ verbundene Fehler während des Kompilierens erkannt. Vor allem bei großen Projekten kann es sonst schwierig werden, einen solchen Fehler im Nachhinein zu identifizieren. Weiters ist der Code mit Type-safety einfacher zu lesen und zu erhalten. Ein weiterer Vorteil ist, dass Code mit Type-safety besser AOT-kompiliert werden kann; Kompilieren von nicht type-safe Code ist zwar möglich, dieser kompilierte Code ist aber weniger effizient\cite{reg422}. \\
